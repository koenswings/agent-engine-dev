### Key design decisions discussed

- `testMode: true` in config.yaml → skips `sudo mount`/`sudo umount` only; all else real
- Disk simulation: create `/disks/<device>/` directory + `touch /dev/engine/<device>`
- App fixtures in `test/fixtures/disk-<app>-v<version>/` with real compose.yaml
- App tests live in app repo (app-kolibri, app-kiwix), copied to disk at build time
- `runAppTests()` hook in harness discovers `tests/` on docked disk and runs it
- `isMinorUpdate(vA, vB)` function needed in Engine (not yet implemented)
- Multi-engine in Proposal B: SSH key must be added during `build-engine` provisioning
- DinD not needed in Proposal B (engine not containerised)

### Diagnostic mode (offline / field)
- `testMode: true` serves double duty: in field, disks are already mounted; skipping mount/umount means sentinel file replay works without physical plug/unplug
- `pnpm test:diagnostic` — discovers real disks, cycles dock/undock via sentinel, smoke tests running apps
- No internet, no image pulls, no fixtures required — works on production Pi
- New open questions: (a) should testMode default ON in Pi builds? (b) should reports be written to disk META.yaml for retrieval?

### Multi-engine app assignment
- Scheme: sorted round-robin — `engines[i % N]` by alphabetical appId
- Deterministic, balanced, zero config
- N > M: idle engines still test startup + mDNS; N < M: concurrent instances (realistic)
- `assignAppsToEngines()` helper in `test/harness/networkDiscovery.ts`
- In diagnostic mode: no assignment — test what's physically present per engine
